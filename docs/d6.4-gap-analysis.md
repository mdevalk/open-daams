# D6.4 v0.5 — gap analysis (spec vs open-daams)

_Snapshot date: 2026-07-24._

A complete gap analysis between **TEHDAS2 D6.4 — DAAMS Technical specification, v0.5 (24 March
2026)** and the current open-daams implementation, section by section. open-daams was originally
built against v0.4 (17 Sept 2025); §7 below records what v0.5 changed, but the focus of this
document is the **total gap versus the current spec**, not just the version delta.

This is TEHDAS2 D6.4-requirement-focused; the plain-EHDS-Regulation-only view is in
[`ehds-gap-analysis.md`](./ehds-gap-analysis.md), the per-article EHDS map is in
[`ehds-article-map.md`](./ehds-article-map.md), and the status tables are in
[`d6.4-status-model.md`](./d6.4-status-model.md).

> **Legend:** ✅ implemented · ◑ partial or simulated · ✗ not implemented · ⏳ blocked on an
> external dependency · — out of DAAMS scope. This is an unofficial community project; nothing
> here is compliance advice.

**Methodology:** every MUST requirement in the section counted once (✅=1, ◑=0.5, ✗=0), simple
average — not effort- or severity-weighted. §11 and §12's requirements are MAY (§12's one
MUST-bundle, R12.0.2, only applies *if* a TDH GUI is built at all, which it isn't) — excluded from
the tally rather than counted as failures. §6's 34 MUST requirements are excluded on the same
basis, but for a scope reason rather than a severity one: §6 (the applicant front office) is the
central platform's / WP6's responsibility, not the back-office DAAMS this project implements — see
CLAUDE.md's scope note. A meaningful share of the remaining misses are **externally blocked**, not
app gaps: the Art. 70 templates don't exist yet, and most of §14 requires a real HealthData@EU/NCP
deployment to test against, which no community implementation can build alone.

## 2. Section-by-section gap

### §6 — Front office (applicant): out of DAAMS scope
D6.4 §6 defines the **applicant-facing front office** — submission UI, applicant portal/login,
document upload, notifications, and dataset browsing/selection. Per this project's own scope (see
CLAUDE.md), that's the **central platform's / WP6's** responsibility, not the back-office DAAMS
open-daams implements — so §6's 34 MUST requirements are excluded from the coverage tally above,
on the same basis as §11/§12's MAY exclusion, just for a scope reason rather than a severity one.

open-daams still ships a staff-facing application form, list, submit/withdraw, and fee estimate as
a practical stand-in so the back-office workflow can be demoed end-to-end — but this is a
convenience shim, not an implementation evaluated against R6.x: there's no applicant login/portal
(an "acting as" role switcher stands in), no document upload (`Document` stays a metadata stub),
no applicant-facing notifications, no draft editing/clone/print/import, and datasets are
free-typed rather than picked from a real catalogue. The **catalogue itself** (Art. 77–80 /
HealthDCAT-AP) and dataset **selection** against it (R6.2.2–6.2.4) are likewise central-platform /
WP6 work; open-daams receives applications with datasets already chosen.

### §7 — Back office (processing): ✅ mostly implemented
Present: role-gated access (R7.1.1, as RBAC), pre-screening completeness check stored as a
structured result (`CompletenessCheck`; R7.2.1–7.2.7), assessment with decision summary + notes +
ethical review (R7.3/7.4), request/receive additional information with deadline void/recalc
(R7.5), and application status updates (R7.6; see `d6.4-status-model.md`).

Gaps:
- **Application flagging + "Restricted Mode"** for sensitive cases (R7.2.8–7.2.10).
- **Multi-country assessment** coordination for HealthData@EU applications (R7.3.6).
- Assessment is free-text rather than a structured per-criterion record (R7.3.2/7.4.2, MAY).

### §8 — Setting due dates: ◑ partial
Present: deadline computed from the submission timestamp with both Art. 68 tracks
(`DecisionTrack` STANDARD/EXPEDITED), voided on info requests and recalculated on resumption, and
an overdue dashboard (R8.0.1, 8.0.2, 8.0.6, 8.0.10).

Gaps:
- **No written justification for a deadline extension** — `deadlineExtended` is a boolean only
  (R8.0.8), and no transmission of that justification cross-border (R8.0.9).
- **No upcoming-due-date notifications** (R8.0.5).
- **No configurable internal timelines** or routing of eligible applications (R8.0.3, 8.0.4).

### §9 — Issuing a decision & permit lifecycle: mixed
- **§9.1 Generating/storing decisions — ◑.** Positive decisions now get a real signed document:
  the 10-section permit PDF is Ed25519-signed on issuance and re-signed on every new version
  ([permit-signing.ts](../src/lib/permit-signing.ts)), the rendered PDF is persisted and
  regenerated whenever its content changes (issuance, authorised-person add/remove, revoke/expire,
  new version — [permit-pdf-store.ts](../src/lib/permit-pdf-store.ts)), and the underlying signed
  JSON document is independently verifiable against `/.well-known/jwks.json` — R9.1.3's "signed,
  time-stamped document" is now met. The signed payload also now carries `grantedDatasets`
  (name + URL, grouped by data holder), so the signature attests to *what* was granted, not just
  the permit's identity and validity window. Remaining gaps: it does **not** use the **Art. 70 Commission
  templates** (R9.1.2, blocked — see §5); **negative decisions get no document at all** — only a
  free-text `decisionSummary`, no signed PDF; and there's no distinct **decision ID** separate from
  the application/permit ID (R9.1.4) — the schema already reserves an `Application.decisionId`
  field for this, but no code populates it.
- **§9.2 Decision cards / decisions pending acceptance — ◑.** A *decision card* is a **structured
  message/record format** (not a UI element) in which the decision is communicated over the
  HealthData@EU interface; a positive decision is issued as an unsigned **pre-permit** for the
  applicant to review, a negative one as a signed decision PDF, and the applicant replies
  Accept/Reject/Appeal (R9.2.1–9.2.4). New in v0.5: for **national** applications too, with a
  **28-day** response window, reminders, and **auto-withdraw → `WITHDRAWN`** on no response
  (R9.2.5–9.2.9). Now wired: the `DECISION_ISSUED` transition
  ([transition/route.ts](../src/app/api/applications/[id]/transition/route.ts)) generates a
  `decisionId`, renders the decision-card PDF (`generate-decision-pdf.ts`), signs it for negative
  outcomes (`signDecisionCard`), and for positive outcomes sets `permitConditionsSentAt` /
  `permitAcceptanceDeadline` (28 days, R9.2.6) / `permitAcceptanceStatus: PENDING`. A new
  `PATCH /api/applications/[id]/decision-card` endpoint lets the applicant Accept/Decline
  (`DecisionCardPanel.tsx`, role-gated to APPLICANT for self-service), gating `PermitPanel`'s
  real-permit issuance on `permitAcceptanceStatus === 'ACCEPTED'`. Still missing: **reminders**
  (R9.2.7–9.2.8 — no notification system exists anywhere in the app) and a **true automatic**
  withdrawal — the 28-day deadline is surfaced as an overdue dashboard item and completed by an
  explicit staff action (`status: 'DECLINED'` on the same endpoint), the same lazy, non-cron
  pattern already used for the equally-unimplemented-as-cron 4-week
  `AWAITING_ADDITIONAL_INFORMATION` no-response case — not a background job that flips the status
  on its own.
- **§9.3 Data permit lifecycle — ✅.** Implemented as a two-axis model: `DataPermitStatus` (5
  permit statuses) + a separate `PermitChangeRequest` (workflow); see `d6.4-status-model.md`. The
  core transition logic is solid: role-gated status changes (R9.3.5), a `DataPermitLog` entry on
  every transition (R9.3.2), and a new signable version + fresh ID on approval (R9.3.6/9.3.7).
  R9.3.6's "generating the up-to-date permit version for signature" is a real, persisted, signed
  document, kept in sync with every mutation that affects it. Version archiving is via the
  `previousPermitId` chain, and the permit detail page ([permits/[id]/page.tsx](../src/app/[locale]/permits/[id]/page.tsx))
  queries every `DataPermit` row sharing the application's id to render a full version history
  (**R9.3.4 all-versions view — correcting a previous mis-assessment in this doc**: it already
  existed, and was mistakenly conflated with a separate one-hop `previousPermit` reference on the
  *application* detail page). The `DP-NL-2025-0001-v2` permit-ID format satisfies all four R9.3.8
  properties (issuing-HDAB identifier, persistent base id, version linkage, chronological
  ordering) — the spec leaves the exact string format to the Member State. Soft spot: `NL` is a
  country code, not a distinct HDAB id, so it would need extending if NL ever ran more than one
  HDAB. **R9.3.9 (delayed effective date for amendments)** is now built: approving an amendment
  with a future effective date creates the new version but keeps it `isCurrent: false` (and SPE
  provisioning re-pointed to it deferred) until a staff member activates it via
  `POST /api/permits/[id]/activate` once the date is due — surfaced as an overdue dashboard item in
  the meantime, the same lazy pattern already used for every other deadline in this app. Renewals
  and revocation appeals stay immediate, per spec.
  Remaining gap:
  - **R9.3.6 transmission.** The signed permit document is available for download, not
    proactively pushed to the data user, data holder, or SPE provider.
- **§9.4 After a decision — ✅.** Amendment and renewal requests are documented, decided, and (on
  approval) issue a new permit version (R9.4.1–9.4.7), via `PermitChangeRequest`.

### §10 — Decision appeals: ◑ partial
Present: an `Appeal` model tracking status/outcome against a decision (R10.0.4/10.0.5 as an
integrated workflow).

Gaps:
- **No display of appeal-mechanism information** to the applicant (R10.0.3).
- **No cross-border appeal handling** via the NCP with receipt confirmation (R10.0.7) — the NCP is
  a mock.

### §11 — Mutual recognition: ✗ not implemented
Fetching already-issued data permits from the central registry to display to HDAB personnel
(R11.0.1) is absent. Spec severity: **MAY**.

### §12 — Trusted data holders (TDH): ✗ not implemented
The simplified-assessment TDH workflow (refer application to a TDH, TDH records assessment +
proposed decision, HDAB accepts/rejects) is absent (R12.0.1–12.0.3). Spec severity: **MAY**, but a
GUI implementation carries MUST sub-requirements. Corresponds to EHDS Art. 72.

### §13 — Interaction with secure processing environments: ◑ / re-scope
v0.5 places SPE **operations** (creation, decommissioning) **outside** D6.4, in deliverable
**D7.4**. The DAAMS's role is limited to (a) handover of permit info to the SPE — permit id +
version, authorised persons, validity, changes — and (b) recording lifecycle events for an
auditable application↔permit↔SPE link (R13.0.1–13.0.2). The app instead models the SPE
**environment** lifecycle (`SpeProvisioningOrder`: REQUESTED→ACTIVE→DECOMMISSIONED), which is now
out of DAAMS scope; the in-scope part (permit-info handover + audit link) is only partially
present via the permit, authorised persons and logs.

### §14 — DAAMS within national EHDS IT infrastructure: ◑ partially real
The design constraint that DAAMS **MUST NOT** communicate directly with the Central Platform
(R14.0.1) is respected. Queue retrieval and decision-card transmission ([ncp-mock.ts](../src/lib/ncp-mock.ts))
now call the real **National Dispatcher OpenAPI** on the HDAB-NL test environment
(`https://ap.hdab-nl.eu/national-dispatcher/{queue,decision-cards}`, R14.0.2/14.0.3) when
`NCP_API_KEY` is configured, falling back to fixed sample data otherwise. Still missing:
queue/retry/logging of failed exchanges (R14.0.4) — a failed call is caught and silently
substituted, not retried or logged for follow-up — and any interoperability test suite against
the NCP (R14.0.5).

## 3. On the term "decision card"

It is **not a UI element**. Per §9.2 it is "the structured format in which decision information is
communicated to applicants via the HealthData@EU Central Platform" — i.e. a standardised
**message/record** (defined against the National Dispatcher OpenAPI) that is materialised as a
document (an unsigned pre-permit PDF for a positive decision; a signed PDF for a negative one). It
is an interoperability construct, closer to a DTO than a screen.

## 4. What v0.5 changed vs v0.4 (context)

| Concept | v0.4 | v0.5 |
|---|---|---|
| Updating application status | §7.6 + §7.7 (per type) | §7.6 (merged) |
| Permit pending acceptance | §9.1 | §9.2 (Decision cards / pending acceptance) |
| Data permit lifecycle | §9.2 | §9.3 |
| No direct link to Central Platform | §13 | §14 |
| SPE | dispersed | §13 (and scope narrowed → D7.4) |

New in v0.5: decision cards + 28-day national auto-withdraw (§9.2), the expanded permit status
set (§9.3), Art. 70 templates/common forms, the HealthData Dispatcher / National Dispatcher
OpenAPI (§14), and the D7.4 SPE reference.

## 5. New references surfaced by v0.5

- **EHDS Art. 70** — Commission templates / EU Common application & request forms (blocks §6.2.5
  and §9.1.2 until the implementing acts land).
- **TEHDAS2 D7.4** — technical specification for Secure Processing Environments.

## 6. Prioritised follow-ups

Buildable now:
1. **Due-date notifications (§8.0.5)** — at least in-app, for case handlers. Also closes the
   remaining §9.2 gap (decision-card reminders) if built as a shared notification mechanism.
2. **Deadline-extension justification (§8.0.8)** — small; add a reason field + flow.

Blocked / optional:
3. **Art. 70 templates** (§9.1.2) — cannot build against non-existent Commission templates; track
   only.
4. **Mutual recognition (§11)** and **trusted data holders (§12)** — spec MAY; build only if the
   demo needs to illustrate them.

Scope decision:
5. **SPE (§13)** — decide whether to keep the (out-of-scope) environment-lifecycle model or refocus
   on the DAAMS-appropriate permit-info handover + audit link.

Out of scope (not open-daams):
- **Front office (§6)** — applicant portal/login, document upload, applicant-facing notifications,
  and the full submission front-end are central-platform / WP6 responsibility; open-daams's
  staff-facing form/list is a demo stand-in only, not an R6.x implementation.
- **Dataset catalogue (Art. 77–80)** and **dataset selection against it (§6.2)** — the catalogue is
  separate work-package-6 work, and selection happens in the central platform / applicant
  front-end; open-daams receives applications with datasets already chosen.
