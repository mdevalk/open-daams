# D6.4 v0.5 status model — application and permit statuses

_Snapshot date: 2026-07-24._

The status tables defined in **D6.4 v0.5 (24 March 2026)**, with how each value maps to
open-daams. Sources: §7.6 Table 1(a) and Table 1(b) (application and decision-phase statuses),
and §9.3 Table 3 (permit statuses). The status values are defined against the **National
Dispatcher OpenAPI specification** (the HealthData@EU interface). Companion to
[`d6.4-gap-analysis.md`](./d6.4-gap-analysis.md).

## Legend

- ✅ present in the app
- ◑ modelled differently / partially (e.g. via a separate field, not as this status)
- ✗ not present

## Application statuses — §7.6 Table 1(a)

| Status value | Label | open-daams |
|---|---|:---:|
| `SUBMITTED` | Submitted | ✅ |
| `PRE_SCREENING` | Pre-screening | ✅ |
| `PROCESSING` | Processing | ✅ |
| `AWAITING_ADDITIONAL_INFORMATION` | Awaiting additional information | ✅ |
| `DECISION_ISSUED` | Decision has been issued | ✅ |
| `WITHDRAWN` | Withdrawn | ✅ |

The app additionally has `DRAFT` (pre-submission / front office, §6). The back-office table
starts at `SUBMITTED` — a full match on the core application lifecycle.

## Decision-phase statuses — §7.6 Table 1(b)

| Status value | Label | open-daams |
|---|---|:---:|
| `POSITIVE_DECISION_ISSUED` | Positive Decision Issued | ◑ `decisionOutcome = POSITIVE` |
| `NEGATIVE_DECISION_ISSUED` | Negative Decision Issued | ◑ `decisionOutcome = NEGATIVE` |
| `POSITIVE_DECISION_ACCEPTED` | Positive Decision Accepted | ◑ `PermitAcceptanceStatus` |
| `NEGATIVE_DECISION_ACCEPTED` | Negative Decision Accepted | ✗ |
| `POSITIVE_DECISION_REJECTED` | Positive Decision Rejected | ◑ `PermitAcceptanceStatus` |
| `NEGATIVE_DECISION_APPEALED` | Negative Decision Appealed | ◑ `Appeal` model |
| `APPEAL_APPROVED` | Appeal Approved | ◑ `Appeal` outcome |
| `APPEAL_REJECTED` | Appeal Rejected | ◑ `Appeal` outcome |

The app does not model these as a single application status; it uses separate fields
(`decisionOutcome`, `PermitAcceptanceStatus`, and the `Appeal` model). Functionally partial —
though as of the decision-card wiring
([transition/route.ts](../src/app/api/applications/[id]/transition/route.ts),
[decision-card/route.ts](../src/app/api/applications/[id]/decision-card/route.ts)),
`PermitAcceptanceStatus` is a live, populated field driving real Accept/Decline behaviour, not
just a reserved-but-unwritten schema value.

## Permit statuses — §9.3 Table 3

| Status value | Description | open-daams |
|---|---|:---:|
| `GRANTED` | Permit has been granted | ✅ |
| `RENEWAL_REQUESTED` | Renewal of permit has been requested | ✗ |
| `RENEWED` | Permit has been renewed | ✅ |
| `RENEWAL_REJECTED` | Renewal request was rejected | ✗ |
| `EXPIRED` | Permit has expired | ✅ |
| `AMENDMENT_REQUESTED` | Amendment to permit has been requested | ✗ |
| `AMENDED` | Permit has been amended | ✅ |
| `AMENDMENT_REJECTED` | Amendment request was rejected | ✗ |
| `REVOKED` | Permit has been revoked | ✅ |
| `REVOKE_APPEALED` | Revocation has been appealed | ✗ |
| `APPEAL_APPROVED` | Appeal was approved | ✗ |
| `APPEAL_REJECTED` | Appeal was rejected | ✗ |

App `DataPermitStatus` holds the **5 permit statuses** (`GRANTED, RENEWED, EXPIRED, AMENDED,
REVOKED`). The other 7 (`*_REQUESTED`, `*_REJECTED`, `*_APPEALED`, `APPEAL_APPROVED`) are
**workflow** statuses and are modelled separately via `PermitChangeRequest` (type × status), so
the ✗ cells above are now covered by that model. See the *permit status vs workflow status*
split — this is the two-axis design the app implements.

## Summary

- **Application core (1a):** full match.
- **Decision phase (1b):** partial — spread across `decisionOutcome` / `PermitAcceptanceStatus` /
  `Appeal` rather than a single status field.
- **Permit lifecycle (Table 3):** the 5 permit statuses in `DataPermitStatus`; the 7 workflow
  statuses are modelled via a separate `PermitChangeRequest` (type × status) whose approval
  drives the permit status. Permit status and workflow status are kept distinct.

Because these values are defined against the National Dispatcher OpenAPI specification, aligning
the app's `DataPermitStatus` (and the decision-phase handling) with them would also aid
cross-border interoperability over the HealthData@EU interface.
