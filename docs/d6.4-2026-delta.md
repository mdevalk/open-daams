# D6.4 update (v0.5, March 2026) ‚Äî delta vs the version open-daams was built on

_Snapshot date: 2026-07-14._

TEHDAS2 published an updated **D6.4 ‚Äî Data Access Application Management System (DAAMS):
Technical specification for health data access bodies**, dated **24 March 2026 (v0.5)**,
incorporating public-consultation feedback. open-daams was built against the earlier
**v0.4 (17 September 2025)**. This note captures what changed and what it means for the app.

> Based on a full read of the v0.5 PDF (74 pp.) against v0.4 (54 pp.). Section numbers below are
> v0.5. Reference URLs and section citations across the repo have been updated to v0.5.

## 2. Substantive changes and implications for open-daams

### 2.1 "Decision cards" + national 28-day auto-withdraw (¬ß9.2) ‚Äî NEW
A **decision card** is the structured format in which decision information is communicated to
applicants via the HealthData@EU Central Platform (positive = unsigned pre-permit PDF for review;
negative = signed decision PDF). New in v0.5: the DAAMS **SHOULD** issue decision cards for
**national** applications too, with a **28-day** window for the applicant to Accept / Reject /
Appeal, reminders, and **automatic withdrawal ‚Üí `WITHDRAWN`** if the applicant does not respond
(R9.2.5‚ÄìR9.2.9).

**Impact:** the app already models a "permit pending acceptance" step (`PermitAcceptanceStatus`),
but does **not** implement the 28-day response deadline or the auto-withdraw. This is a concrete
new behaviour to consider.

### 2.2 Richer data-permit status set (¬ß9.3)
v0.5 defines twelve permit statuses (Table 3): `GRANTED`, `RENEWAL_REQUESTED`, `RENEWED`,
`RENEWAL_REJECTED`, `EXPIRED`, `AMENDMENT_REQUESTED`, `AMENDED`, `AMENDMENT_REJECTED`, `REVOKED`,
`REVOKE_APPEALED`, `APPEAL_APPROVED`, `APPEAL_REJECTED`.

**Impact ‚Äî IMPLEMENTED (two-axis model).** The app keeps `DataPermitStatus` as the five permit
statuses and models the 7 workflow values (`*_REQUESTED`/`*_REJECTED`/`*_APPEALED`/`APPEAL_APPROVED`)
via a separate `PermitChangeRequest` (type ‚àà {AMENDMENT, RENEWAL, REVOCATION_APPEAL} √ó status ‚àà
{REQUESTED, APPROVED, REJECTED}); approving a request drives the permit status change and issues a
new version. See `lib/permit-change.ts`, `api/permits/[id]/change-requests/*`, and
`PermitChangeRequestPanel`. Also relevant:
R9.3.8 prescribes a structured permit-ID format (issuing HDAB + persistent base + version +
chronology, SemVer-like) ‚Äî the app's `previousPermitId` + `DP-NL-‚Ä¶` numbering partially matches. See
[`d6.4-status-model.md`](./d6.4-status-model.md) for all three status tables mapped to the app.

### 2.3 Commission templates / EU Common forms (Art. 70) ‚Äî NEW article reference
Decisions must use the **templates developed by the Commission under Article 70 EHDS** (R9.1.2),
and the application/request forms must match the **EU Common Health Data Access Application /
Request forms**, to be further specified by implementing acts under Art. 70 (R6.2.5/R6.2.7).

**Impact:** the app generates its own permit PDF and uses a custom form. **Art. 70** is a
reference that was not in our article map and is worth tracking once the Commission templates
land.

### 2.4 Dataset selection from the National Health Dataset Catalogue (¬ß6.2)
Applications must be created for one or more datasets **described in the National Health Dataset
Catalogue** (R6.2.2), optionally via a "shopping cart" metaphor, with read-only dataset
descriptions shown (R6.2.4).

**Impact:** reinforces the existing **biggest gap** ‚Äî there is no dataset catalogue in the app
(EHDS Art. 77‚Äì80; see `gap-analysis.md`), so datasets are free-text rather than selected from a
catalogue.

### 2.5 SPE scope narrowed (¬ß13) ‚Äî points to D7.4, not D7.1
v0.5 states SPEs are **outside the operational scope** of D6.4 and defined separately in
**TEHDAS2 deliverable D7.4** on SPEs. The DAAMS's SPE role is limited to (a) handover / structured
availability of permit info to the SPE (permit id + version, authorised persons, validity,
changes) and (b) recording lifecycle events for an auditable application‚Üîpermit‚ÜîSPE link
(R13.0.1‚ÄìR13.0.2).

**Impact:** the app's `SpeProvisioningOrder` models the *environment* lifecycle
(`REQUESTED ‚Üí ACTIVE ‚Üí DECOMMISSIONED`), which v0.5 places **outside** DAAMS scope. The
DAAMS-appropriate part is the permit-info handover + audit link. Also note the SPE technical spec
is **D7.4** (D7.1 remains the how-to-use-data guideline).

### 2.6 Cross-border interface (¬ß14) ‚Äî HealthData Dispatcher / National Dispatcher OpenAPI
v0.5 formalises the NCP gateway as the **HealthData Dispatcher**, with statuses and operations
defined in a **National Dispatcher OpenAPI specification**, and restates that the DAAMS **MUST
NOT** communicate directly with the Central Platform (R14.0.1).

**Impact:** consistent with the app's design note (all cross-border traffic via the NCP); the app
simulates this with a mock NCP queue.
